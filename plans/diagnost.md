# Фича: «Диагност» (по мотивам github.com/adam-s/car-diagnosis) + PWA-полировка

## Цель

Владелец описывает симптом (и/или записывает звук на телефон) → структурированный триаж: ранжированные причины с уверенностью, светофор срочности, что проверить самому, что писали владельцы Орландо про такое. Из car-diagnosis берём философию (триаж, честное «не уверен»), не Python/CLAP-модель — движок: LLM + RAG по чату сообщества + Gemini для звука.

## Скоуп MVP

**Входит:** текстовый триаж (`/api/diagnose`) + UI со светофором; запись с микрофона (MediaRecorder, 10–15 сек) → Gemini 2.5 Flash через OpenRouter; кросс-чек с RAG по чату; PWA: рабочие шорткаты `?action=`, шорткат «Диагностика», офлайн-страница.

**НЕ входит:** порт CLAP-модели / свой ML; загрузка файлов и YouTube-ссылок; OBD-коды; платежи/авторизация (диагност публичный, как «Спросить»).

## Фазы

### Фаза 1: Каркас + текстовый триаж (API) ✅
**Файлы:** `plans/diagnost.md`, `CLAUDE.md`, `lib/diagnose.ts`, `app/api/diagnose/route.ts`.
- `lib/diagnose.ts`: триаж-промт → DeepSeek через `chat()` из `lib/openrouter.ts`. Строгий JSON: `{causes:[{name,confidence,why}], severity:"ok|soon|urgent", checks:[], uncertain, questions:[]}`. Мало данных → `uncertain:true` + уточняющие вопросы вместо угадывания.
- Кросс-чек: эмбеддинг симптома (Mistral) → треды сообщества → `sources` (тип `SearchResult`, фронт уже умеет).
**Критерий:** «стук спереди на мелких кочках» → валидный JSON с причинами и тредами; «машина плохая» → `uncertain:true` с вопросами.

### Фаза 2: UI «Диагност» ✅
**Файлы:** `app/page.tsx`.
Секция «🩺 Диагност»: поле симптомов → карточки причин (уверенность бейджем), светофор (🟢 можно ехать / 🟡 к мастеру / 🔴 не езди), «что проверить самому», треды сообщества. При `uncertain` — вопросы чипами (нажал → дописалось в симптом). Дисклеймер «триаж, не вердикт мастера».
**Критерий:** полный цикл в превью (десктоп + мобильная ширина): симптом → триаж → уточнение → новый триаж.
**Промт:** «Фаза 2 из plans/diagnost.md: UI Диагноста в app/page.tsx в стиле существующих карточек (shadow-neu, rounded-3xl). Открой превью, проверь мобильную ширину, почини что упало.»

### Фаза 3: Звук — запись и аудио-анализ ✅ (+ история машины как фильтр вероятностей)
**Файлы:** `lib/diagnose.ts`, `app/api/diagnose/route.ts`, `app/page.tsx`.
- Фронт: MediaRecorder → webm/opus (iOS даст mp4/aac), лимит 15 сек, индикатор, прослушать перед отправкой.
- Бэк: аудио base64 как `input_audio` в Gemini 2.5 Flash через OpenRouter → описание звука → в тот же триаж-промт.
- Фолбэк: аудио не принялось → честная ошибка «опиши словами», текстовый путь не ломается.
**Критерий:** записанный звук (например, с ролика «stuck hydraulic lifter») даёт осмысленное описание в ответе.
**Промт:** «Фаза 3 из plans/diagnost.md: MediaRecorder на фронте + аудио-ветка в diagnose через Gemini 2.5 Flash. Проверь в превью с реальным микрофоном, почини что упало.»

### Фаза 4: PWA-полировка ✅
**Файлы:** `public/manifest.webmanifest`, `public/sw.js` (v8), `app/page.tsx`, офлайн-страница.
- Починить мёртвые шорткаты: страница читает `?action=log|ask|diagnose` и открывает секцию.
- Шорткат «Диагностика» в манифест; офлайн-фолбэк вместо ошибки браузера.
**Критерий:** Lighthouse PWA зелёный; `?action=diagnose` открывает секцию; offline-режим показывает офлайн-страницу.
**Промт:** «Фаза 4 из plans/diagnost.md: обработка ?action=, шорткат Диагностика, офлайн-фолбэк, SW v8. Проверь в превью offline-режим и ?action=diagnose, почини что упало.»

## Риски

- **Аудио через OpenRouter:** webm/opus у Gemini может капризничать → пробуем mp4/aac от MediaRecorder. Проверяем в Фазе 3 первым делом.
- **iOS + микрофон в PWA:** getUserMedia в standalone с iOS 14.3+, нужен жест пользователя — кнопка записи покрывает.
- **Стоимость:** лимит записи и символов, без стриминга; rate-limit — после MVP.
- **Ответственность:** триаж, не диагноз; красный светофор всегда советует мастера.
